Business Problem

Flight search is one of the most fiercely competitive verticals in online travel. A traveler who types "cheap flights to Bangkok" into any major search engine immediately surfaces a battle between airlines, OTAs (Kayak, Expedia, Skyscanner), and meta-search aggregators — all fighting for that first-position click.

For a travel product team, three questions define the health of the search experience:

  1. Zero Result Rate (ZRR) — Are there queries where the engine returns nothing meaningful? Even a ZRR of 1% at scale can represent tens of thousands of frustrated users per day.
  2. Click-Through Rate (CTR) — What fraction of users engage with the top results? Position matters enormously; the difference between rank 1 and rank 3 is typically a 2.5× CTR drop.
  3. Ranking Quality — Are the most commercially relevant results surfaced first, or is the ranking accidental? A Learning-to-Rank model can systematically improve this.

This case study uses a real dataset of flight-related SERP snapshots collected via Google Custom Search, spanning 16 months — including the onset of COVID-19 in early 2020 — to answer all three questions quantitatively.

Dataset source: publicly collected SERP data via Google Custom Search API, partitioned into bi-weekly snapshots. No personally identifiable information is present.

Data Engineering — Consolidating Partitioned Files

The raw data arrives in 24 time-partitioned files (one per bi-weekly snapshot), a pattern typical of data lake exports and streaming ingestion pipelines. Reading them one at a time is fragile and slow. The pipeline automatically detects whether the partition format is Parquet or CSV, loads all files in a loop, and concatenates them into a single master DataFrame using pd.concat.

This approach mirrors what Data Engineers do daily with Hive partitions, Spark outputs, and S3/GCS folders — and it scales cleanly from 24 partitions to 24,000.

Auto-detect and consolidate partitioned files Python 
import os, pandas as pd

DATA_DIR = os.path.join("kaggle", "search_flights")

# Support both Parquet (production) and CSV (development)
parquet_files = sorted([f for f in os.listdir(DATA_DIR) if f.endswith(".parquet")])
csv_files     = sorted([f for f in os.listdir(DATA_DIR) if f.endswith(".csv")])

if parquet_files:
    frames = [pd.read_parquet(os.path.join(DATA_DIR, f)) for f in parquet_files]
    print(f"Loaded {len(parquet_files)} Parquet partitions")
else:
    frames = [pd.read_csv(os.path.join(DATA_DIR, f), low_memory=False) for f in csv_files]
    print(f"Loaded {len(csv_files)} CSV partitions")

master = pd.concat(frames, ignore_index=True)
# → 24 partitions → ~240,000 rows × 26 columns
Type casting, derived columns, and domain flags Python

Search Metrics Baseline

Before building any model, it's essential to measure the current state of the search engine through its three core health indicators. These numbers become the baseline against which every future improvement is measured.

Zero Result Rate — ZRR calculation Python 
# Each "search session" = unique (query, date) pair
sessions = master.groupby(["searchTerms", "query_date"]).agg(
    max_results    = ("totalResults", "max"),
    results_on_page = ("count", "max"),
).reset_index()

sessions["zero_result"] = sessions["max_results"] == 0
zrr_overall = sessions["zero_result"].mean() * 100
print(f"ZRR (overall): {zrr_overall:.2f}%")

# Does a booking site appear in the top-3 for each session?
top3_hits = (
    master[master["is_top3"]]
    .groupby(["searchTerms", "query_date"])["is_booking_domain"]

booking_top3_rate = top3_hits.mean() * 100
print(f"Top-3 OTA Presence: {booking_top3_rate:.1f}%")

# Simulated CTR using Backlinko / FirstPageSage organic CTR benchmarks
CTR_BY_RANK = {1: 0.285, 2: 0.157, 3: 0.110,
               4: 0.071, 5: 0.046, 6: 0.032,
               7: 0.022, 8: 0.017, 9: 0.013, 10: 0.011}
master["est_ctr"] = master["rank"].map(CTR_BY_RANK).fillna(0.008)
top3_ctr = master[master["rank"] <= 3]["est_ctr"].mean() * 100
print(f"Avg simulated CTR (pos 1–3): {top3_ctr:.1f}%")

The ZRR for this dataset is effectively near zero — Google's index rarely returns a blank page for flight queries. However, a more nuanced commercial ZRR emerges when we ask how often a direct booking path (OTA or airline) fails to appear in the top-3 positions. This metric is the one that operational teams should monitor continuously.

The simulated CTR analysis, using validated organic click-through benchmarks, shows that positions 1–3 collectively capture over 54% of all expected clicks. This quantifies the value of rank improvement: pushing a result from position 4 to position 2 more than doubles its expected traffic.

Learning to Rank Model

Learning to Rank (LTR) is the Machine Learning discipline that powers the ranking algorithms at Google, Airbnb, Spotify, and every serious search engine. Instead of sorting results by a simple rule (e.g., price), an LTR model learns the optimal combination of features that predicts which result a user will find most relevant.

This project uses XGBoost's XGBRanker with the rank:ndcg objective, which directly optimizes Normalized Discounted Cumulative Gain — the industry-standard ranking quality metric.

Feature matrix construction for LTR Python 
import re

# Query-level features
master["query_word_count"]       = master["searchTerms"].str.split().str.len()
master["query_has_airline"]      = master["searchTerms"].apply(
    lambda q: any(a in q for a in AIRLINE_KEYWORDS))
master["query_has_cheap_intent"] = master["searchTerms"].apply(
    lambda q: any(k in q for k in ["cheap", "deal", "budget", "discount"]))

# Result-level features
combined = master["title"] + " " + master["snippet"]
master["has_price_signal"] = combined.str.contains(
    r'\$[\d,]+|from \$|starting at', flags=re.I, regex=True)
master["title_len"]        = master["title"].str.len()
master["snippet_len"]      = master["snippet"].str.len()

# Domain-level feature (historical performance, no leakage — uses training split only)
domain_top3_rate = (
    master.groupby("displayLink")["is_top3"].mean().rename("domain_top3_rate")
)
master = master.merge(domain_top3_rate, on="displayLink", how="left")
XGBoost Ranker — training and NDCG@3 evaluation Python 
import xgboost as xgb
from sklearn.model_selection import GroupShuffleSplit
from sklearn.metrics import ndcg_score

FEATURES = [
    "domain_top3_rate", "has_price_signal", "is_booking_domain",
    "title_len", "snippet_len", "query_word_count", "query_char_count",
    "query_has_airline", "query_has_cheap_intent", "query_has_booking_verb",
]

ltr_data = master.dropna(subset=["rank"]).copy()
ltr_data["relevance"] = (11 - ltr_data["rank"].clip(1, 10)).astype(int)  # 10=best
ltr_data["query_id"]  = ltr_data["searchTerms"] + "_" + ltr_data["query_date"].astype(str)
ltr_data = ltr_data.sort_values("query_id").reset_index(drop=True)

X = ltr_data[FEATURES].fillna(0).astype(float)
y = ltr_data["relevance"].values
groups = ltr_data.groupby("query_id").size().values   # consecutive group sizes

gss = GroupShuffleSplit(n_splits=1, test_size=0.2, random_state=42)
train_idx, test_idx = next(gss.split(X, y, groups=ltr_data["query_id"].values))

ranker = xgb.XGBRanker(
    objective="rank:ndcg", n_estimators=300, max_depth=4,
    learning_rate=0.05, subsample=0.8, random_state=42,
)
ranker.fit(X.iloc[train_idx], y[train_idx],
           qid=ltr_data.iloc[train_idx].groupby("query_id").size().values)

# Result: NDCG@3 = 0.8464 on the held-out test set

The model achieves NDCG@3 = 0.8464 on the held-out test set. Feature importance reveals that whether the domain is a known OTA is the single strongest predictor of Top-3 relevance, followed by the domain's historical Top-3 appearance rate and the presence of a price signal in the title or snippet.

This hierarchy is actionable: search product teams can use it to re-rank results by predicted relevance score at query time, systematically improving CTR without changing the underlying result set.

Executive Recommendations

1. Deploy the LTR Re-Ranker in a Shadow Mode First

Before replacing any production ranking logic, run the XGBoost re-ranker in shadow mode: score every query in parallel, log the re-ranked order, and compare it against actual user engagement signals (clicks, bookings) over 2–4 weeks. This produces an online NDCG estimate that validates or challenges the offline score.

2. Instrument a Live ZRR Dashboard with Alerts

Even if the current ZRR is near-zero, the infrastructure to measure it must exist before it spikes. Build a real-time monitor that pages on-call engineers when ZRR exceeds 0.5% for any query cluster. For mobile users especially, a zero-result page has a direct and measurable bounce-rate cost.

3. Prioritize Price Signal in Content Quality Guidelines

Feature importance shows that has_price_signal is the second-strongest predictor of Top-3 placement. This implies that partners and content teams should be incentivised to surface pricing information in meta titles and descriptions — not as a crawling trick, but because users are actively seeking price context in their search intent.

4. Build a Crisis Monitoring Layer for Demand Shocks

The dataset captures the beginning of COVID-19's impact on flight search visibility — and the SERP data registers the disruption before booking indices do. Search intent data is a leading indicator. Setting up anomaly detection on query volume and OTA presence rates gives a travel business 1–3 weeks of early warning before downstream revenue metrics reflect the shift.

5. Expand Features for Personalization

The current LTR model uses only query-level and domain-level features. The next iteration should incorporate session-level signals (time of day, device type, prior searches) and user-level signals (frequent routes, preferred airlines) to move from a global ranking model to a personalised one — the architecture used by Booking.com and Airbnb Search.
