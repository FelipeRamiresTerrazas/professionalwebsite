PedidosYa App Friction Analysis: 3,438 Real Reviews Across BR & LATAM ES

A multi-market Product Operations pipeline that scrapes 3,438 live Google Play Store negative reviews for PedidosYa (Delivery Hero) across Brazil (PT) and LATAM ES (Bolivia + Argentina), applies language-aware NLP keyword analysis per market, and surfaces a sharp divergence in pain-point hierarchy: technical instability dominates BR while financial trust failure dominates LATAM ES — two different root causes hiding behind one product.

Executive Summary

Rather than relying on internal analytics or survey panels, this pipeline mines the publicly available signal in 3,438 real Google Play Store negative reviews (1‑2★) across two language markets (April 2026). Reviews were scraped using separate filter_score_with=1 and filter_score_with=2 passes to maximise corpus depth, then deduplicated by reviewId. The NLP pass uses language-matched stopwords — Portuguese for BR, Spanish for LATAM ES — ensuring organic term frequency reflects genuine signal, not stopword noise.

Product Ops Recommendations

Five structured recommendations derived from the multi-market friction signals. Each card maps a quantified data point to a revenue/retention impact hypothesis and a concrete team action.

💥 BR — App Stability is the #1 Conversion Killer

Problem Found
22.9% of BR negative reviews (330 mentions) cite crashes, freezes, or the app failing to open. “abre” is the #1 organic word in the BR corpus — users can’t get past the launcher screen.

Impact Hypothesis
Users in the consideration phase hit a fatal error before reaching a restaurant menu: zero impressions served, zero GMV, full churn with no recovery path.

Recommended Action
Enforce a Crash-Free Sessions SLA gate (≥ 99% CFR) before any new feature ships. Instrument Crashlytics alerting at ≥ 0.5% crash rate. Establish a Platform Stability squad with release veto power.

📍 BR — GPS/Address Failures Block the Entire Discovery Funnel

Problem Found
127 BR negative reviews (8.8%) reference GPS or address-validation failures. The app cannot correctly detect user location or validate a manually entered address.

Impact Hypothesis
Address failure is a first-mile block: without a valid location the app cannot show relevant restaurants or estimate fees, collapsing the full discovery funnel.

Recommended Action
Integrate redundant geocoding fallback (Google Maps + HERE). Instrument address auto-confirm rate as a leading funnel KPI. Ship saved-addresses for returning users to bypass GPS dependency.

⚔️ BR — iFood is a Measurable, Active Churn Destination

Problem Found
iFood is explicitly named 66 times in BR negative reviews (organic term, not prompted) — a direct churn destination that users name voluntarily in review text.

Impact Hypothesis
Each iFood mention represents a user who has already evaluated the alternative and published their decision. Word-of-mouth churn multiplies retention cost and poisons organic acquisition channels.

Recommended Action
Monitor iFood mention velocity as a leading churn KPI in weekly dashboards. Launch a win-back flow targeting users dormant ≥ 21 days. Benchmark UX patterns against iFood quarterly.

💸 LATAM ES — Payment Failures Destroy Maximum-Intent Moments

Problem Found
29.8% of LATAM ES negative reviews (595 mentions) cite payment failures: double-charges, card declines, coupon non-application, and refund loops. Top organic terms are dinero, tarjeta, cobrar, pago.

Impact Hypothesis
A user who has browsed, selected items, and entered checkout has maximum purchase intent. Payment failure here yields 100% GMV loss from a warm, committed buyer and turns a loyal user into a vocal critic.

Recommended Action
Audit gateway success rate per PSP (Bolivia: QR/mobile wallet; Argentina: tarjeta + Mercado Pago). Introduce instant self-service refund flow (no agent required) — this alone would deflect the bulk of the Support/atención volume (14.4%) in ES markets.

💬 All Markets — No Market Has a Functional Self-Serve Recovery Path

Problem Found
Support/atendimento appears in the top-3 friction signals in both language markets — 8.8% in BR (127 mentions), 14.4% in LATAM ES (289 mentions). Users consistently describe an experience where no recovery path exists after a bad order.

Recommended Action
Implement an in-app “What went wrong?” micro-survey at the 24-hour post-order mark. Surface a one-tap self-service resolution flow for the top 3 complaint types (wrong/missing item, payment failure, estimated time exceeded) before routing to a human agent.

Data Pipeline

The multi-market pipeline runs two Python scripts with a JSON contract at the seam between the data layer and the NLP layer. Reviews are scraped in two separate score bands per market to maximise corpus depth, then analysis uses language-matched stopword sets.

01
🕷 Scraper — scrape_delivery_hero_global.py

Targets com.pedidosya per market (BR: lang=pt; BO/AR: lang=es). Fetches filter_score_with=1 and filter_score_with=2 in separate passes (200-row batches, up to 60 calls per band), then deduplicates by reviewId. Output: reviews_delivery_hero_global.json.

02
🧠 NLP Analysis — analyze_voc_global.py

Loads each market’s corpus, applies language-matched stopwords (PT for BR, ES for LATAM ES), runs bilingual regex PATTERNS, and extracts top-20 organic terms via Counter. Output: voc_analysis_results.json.

03
📊 Visualisation — HTML / Matplotlib

Cross-market bar charts rendered directly in HTML from the JSON results. Detailed BR figures (score distribution, keyword frequency, organic terms, funnel) generated by Matplotlib dark-theme at 160 DPI. Output: this page + 4 PNG files in figures/.

scrape_delivery_hero_global.py — dual-band scrape + dedup Python 
TARGET_NEG = 2000
MAX_CALLS  = 60
BATCH      = 200

def scrape_score_band(app_id, lang, country, score, label, target):
    collected, token = [], None
    for _ in range(MAX_CALLS):
        if len(collected) >= target:
            break
        result, token = reviews(
            app_id, lang=lang, country=country,
            sort=Sort.NEWEST, count=BATCH,
            filter_score_with=score,
            continuation_token=token,
        )
        if not result:
            break
        collected.extend(result)
        if token is None:
            break
        time.sleep(0.4)
    return collected

def scrape_market(app_id, lang, country, label, target=TARGET_NEG):
    per_band = (target + 1) // 2
    band1 = scrape_score_band(app_id, lang, country, 1, label, per_band)
    band2 = scrape_score_band(app_id, lang, country, 2, label, per_band)
    seen, merged = set(), []
    for r in band1 + band2:
        if r["reviewId"] not in seen:
            seen.add(r["reviewId"])
            merged.append(r)
    return merged
analyze_voc_global.py — language-aware PATTERNS pass Python 
MARKET_STOPWORDS = {
    "pedidosya_BR": [STOP_PT, STOP_EN],   # Brazilian Portuguese
    "pedidosya_BO": [STOP_ES, STOP_EN],   # Spanish (Andean)
    "pedidosya_AR": [STOP_ES, STOP_EN],   # Spanish (Andean)
    "foodpanda_BD": [STOP_EN],            # English
}

def analyze_market(reviews_list, patterns, stopword_sets, label):
    texts = [str(r.get("content") or "").lower() for r in reviews_list]
    stopwords = set().union(*stopword_sets)

    freq = {kw: sum(bool(re.search(pat, t)) for t in texts)
            for kw, pat in patterns.items()}

    all_words = " ".join(texts).split()
    top_words = Counter(
        w.strip(".,!?;:()") for w in all_words
        if len(w) > 3 and w not in stopwords
    ).most_common(20)
    return {"total": len(texts), "freq": freq, "top_words": top_words}
